HA Wordpress Site (2) - EC2 setup

Building a fault tolerant Wordpress site: LAB 2 - Setting Up EC2

Checking EC2

Go to EC2 > Instances

  1. Grab the public IP address of your EC2 (created in part 1)
  1. ssh to EC2
    ssh ec2-user@<ip> -i <mykey.pem>
  1. Try to access html directory. If you can access there apache has been installed correctly.
    cd /var/www/html
  1. Check wordpress files are there using ls
    ls
  1. Check that .htaccess has been installed:
    cat .htaccess
  1. Check that apache service has been started:
    service httpd status

    If it's not showing a green "active (running)", try to start typing: service httpd start

Installing Wordpress

  1. Grab your public ip address and put it in the browser, you should see something like the figure below. Click on Let's go! button.
  1. Fill the database information below:
  1. If the page below appears, copy the code on your clipboard, ssh to EC2 instance and paste it manually.
    ssh ec2-user@<ip> -i <mykey.pem>
cd /var/www/html
nano wp-config.php

    Paste the code, hit CTRL+X, hit Y, hit Enter (save the file)

    Go back to the web browser, click on run the installation.

  1. Install Wordpress! Fill the data and click on "Install WordPress"
  1. Wordpress is Installed! You should see the following. Click on Log In button and continue!

Login into Wordpress and create a post

  1. Login to Wordpress using your credentials. If you want to access to login again, just write in your browser: <ip>/wp-login.php
  1. Write a new post with a Hello world and add 2 pictures.
  1. Click on Publish button (top-right), and then on View post.

Check the media (pictures) uploaded in Wordpress

  1. Go back to your Terminal window, ssh into your EC2 instance as before if you are not there yet, and cd to /var/www/html. Then:
    cd wp-content

    There are your Wordpress pictures! But they are being stored in EC2... if EC2 is terminated, this pictures will be lost. Let's add more resiliance to this poor images.

Store Wordpress uploaded files to S3 for redundancy

  1. Go back to html directory:
    cd /var/www/html
  1. Make sure that the EC2 - S3 Role is working:
    aws s3 ls
  1. Copy the files to S3 media bucket:
    aws s3 cp --recursive /var/www/html/wp-content/uploads s3://acloudguruwp-media-123090983

Store full copy of Wordpress on S3

If we lose our EC2, we want a full copy of our wordpress on S3. To do so:

  1. Copy files to S3 bucket (the code one)
    aws s3 cp --recursive /var/www/html s3://acloudguruwp-code-12351345
  1. Check that the files are correctly saved:
    aws s3 ls s3://acloudguruwp-code-12351345
  1. Check that we also have our healthy.html from our bootstrap script:
    cd /var/www/html
ls
cat healthy.html

Force Wordpress to serve uploaded files using Cloudfront rather than using EC2 images

  1. Check your .htaccess file and edit it. In .htaccess we've got what's called a URL rewrite rule. This essentially allows our website to instead of serving the images out of EC2, it's going to serve our images out of CloudFront.
    cd /var/www/html
nano .htaccess

    This rewriterule is saying, rewrite this URL (wp-content/uploads/(.*)$ to this new URL → http://<cloudfrontURL>/$1 [r=301,nc]

  1. Edit the current cloudfront URL and put our Cloudfront URL (find it in Cloudfront > Click on our Distribution > General tab > Domain Name :
    rewriterule ^wp-content/uploads/(.*)$ http://<cloudfront-url>/$1 [r=301,nc]

    Save it: CTRL+X → Y → Enter

  1. Make sure your S3 bucket is up to date with the latest changes made:
    aws s3 sync /var/www/html s3://acloudguruwp-code-12351345
  1. This still won't work. We have to tell Apache that we're going to allow URL rewrites. In order to do that:
    cd /etc/httpd
ls
cd conf
ls
  1. Copy first your file (backup):
    cp httpd.conf httpd-copy.conf
  1. Edit httpd.conf:
    nano httpd.conf
  1. Scroll down until you find "AllowOverride None"
  1. Change AllowOverride None to AllowOverrideAll
  1. Restart the apache service:
    service httpd restart
  1. Now go to your aws console → S3 → click on acloudguruwp-media-123090983 bucket → Configuration → Permissions → Bucket Policy, copy and paste this:
    {
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": [
        "s3:GetObject"
        ],
      "Resource": [
        "arn:aws:s3:::BUCKET_NAME/*"
        ]
    }
  ]
}

    Change BUCKET_NAME for your ARN, and hit Save before leaving. You should see a warning telling that "This Bucket has public access". This is what we want.

  1. If an error appears, Go back to your list of buckets, click on your bucket acloudguruwp-media-123090983 → Edit public access settings → make sure everything is unchecked and hit save. Then do step 10 again.

Check that Wordpress post is getting media from Cloudfront + S3

  1. Go to your wordpress post edit page → Right click → Copy image → Paste the URL in the browser and check the URL.

Create an Application Load Balancer and put EC2 behind it

  1. Go to EC2 > Load Balancing > Load Balancers > Create Load Balancer
  1. Step 1. Configure Load Balancer
  1. Step 2: Configure security Settings → Next
  1. Step 3: Configure Security Groups
  1. Step4: Configure routing
  1. Step5: Register targets. Remember to click on the "Add to registered" button!! (otherwise you will need to do it later).
  1. Step 6: Review → Hit Create
  1. Our ALB is created! Wait until deployed (3-4 minutes)

Route 53 - Point our Domain to our ALB

Go to Route 53 > Hosted Zones > Click on your domain name

  1. Create a record set (a type A, with alias target the new ALB created).

3. Go to your browser and try to put your domain name! (wait a bit, it can take up to 1 day or more for DNS to propagate).
