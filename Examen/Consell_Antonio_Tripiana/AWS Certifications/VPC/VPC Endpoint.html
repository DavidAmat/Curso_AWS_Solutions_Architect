VPC Endpoint

What is a VPC Endpoint?

A VPC Endpoint allows AWS services to connect to each other without leaving AWS and without requiring internet.

A VPC Endpoint enables you to privately connect your VPC to supported AWS services and VPC endpoint services powered by PrivateLink without requiring an internet gateway, NAT device, VPN connection or AWS Direct Connect connection. Instances in your VPC do not require public IP addresses to communicate with resources in the service. Traffic between your VPC and the other service does not leave the Amazon network.

Endpoints are virtual devices. They are horizontally scaled, redundant and highly available VPC components that allow communication between instances in your VPC and services without imposing availability risks or bandwidth constraints on your network traffic.

Types of VPC Endpoints

  • Interface Endpoints

    An interface endpoint is an Elastic Network Interface (ENI) with a private IP address that serves as an entry point for traffic destined to a supported service.

    • The following services are supported:
      • API Gateway, CloudFormation, CloudWatch, CodeBuild, Config, EC2 API, Elastic Load Balancing, Key Management Service, Kinesis, SageMaker, Secrets Manager, Security Token Service, Service Catalog, SNS, SQS Systems Manager
      • Endpoint services hosted by other aws accounts
      • Supported AWS Marketplace partner services
  • Gateway Endpoints

    They are quite similar to endpoints, but acting as a Gateway. Services supported at the moment are: Amazon S3, DynamoDB

VPC Endpoints in action

Our current VPC environment looks like the figure below. If we want to send a file to S3 from our private servers, we would need to reach our NAT Gateway, then go to public internet and then store the file in our S3 Buckets using S3 public endpoints (this is because S3 server cannot be in our VPC).

To reach the S3 bucket without leaving VPC, we can use a VPC Endpoint, that will connect our private instance directly to S3 via internal connection (without leaving AWS network and not accessing internet)

Create a VPC Endpoint

Prerequisites:

  1. IAM Role for EC2 service with S3 admin access policy.
  1. EC2 launched and associated with the new IAM Role created in 1.
  1. You might encounter problems with NACLs, so its better if you disable it for this test purpose (simply go to NACL service and move the private subnet to the main NACL (open access one).
  1. SSH forwarding using a bastion host to SSH into the private instance. Alternatively (although is not recommended) you can send your .pem file to the public instance and then ssh to the private instance from the public instance using this .pem file.
  1. NAT Gateway disabled (or route table disabled to check that private connections goes through VPC endpoint).

First, let's create a new VPC Endpoint:

Go to VPC > Endpoints > Create endpoint

  • Select the s3 Gateway (as shown in figure)
  • VPC ‚Üí Select your VPC
  • Route ‚Üí main route table (default one, private connection only)
  • Policy ‚Üí Full Access

Connect to your private instance using ssh and bastion host, and then use the following command to check if your EC2 instance is connected to S3 via VPC Endpoint (if you don't specify the region it can fail):

aws s3 ls --region us-east-1

Note that the region can be different in your case (you can check where are your buckets)
